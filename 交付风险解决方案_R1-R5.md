## R1｜原生客户端零编译验证

### 问题回顾
`app/` 下的 CMP+KMP 客户端源码从未编译过，本机无 JDK/Android SDK，Xcode 仅命令行工具。iOS `AudioCapture.ios.kt` 的 AVAudioConverter cinterop 风险最高。

### 目标
1. 本机能跑通 `:composeApp:assembleDebug`（Android）与 `linkDebugFrameworkIos*`（iOS framework 编译）。
2. CI 每次推送自动编译三端，编译失败即红灯，杜绝"纸面完成"。

### 工具/资源清单
| 工具 | 版本 | 用途 | 安装方式 |
|------|------|------|---------|
| Homebrew | 5.x | 包管理 | 本机已装 |
| Temurin JDK | 17 | Kotlin/Gradle 运行时 | `brew install --cask temurin@17` |
| Android Studio | Ladybug+ | Android SDK + KMP 插件 | `brew install --cask android-studio` |
| Android SDK | API 35 | 编译目标 | Android Studio SDK Manager |
| Xcode | 16+ | iOS framework + 真机 | App Store（**人工安装，约 12GB**） |
| Gradle | 8.9 | 由 wrapper 自动拉取 | 无需手动装 |

### 操作步骤（本机）

**步骤 1：安装 JDK 17 并验证**
```bash
brew install --cask temurin@17
# 写入 shell 配置（zsh）
echo 'export JAVA_HOME=$(/usr/libexec/java_home -v 17)' >> ~/.zshrc
source ~/.zshrc
java -version   # 期望输出 openjdk version "17.x"
```

**步骤 2：安装 Android SDK**
```bash
brew install --cask android-studio
# 打开 Android Studio → More Actions → SDK Manager
# 勾选：Android SDK Platform 35、Android SDK Build-Tools 35、Android SDK Command-line Tools
# 完成后写入环境变量：
echo 'export ANDROID_HOME=$HOME/Library/Android/sdk' >> ~/.zshrc
echo 'export PATH=$PATH:$ANDROID_HOME/platform-tools' >> ~/.zshrc
source ~/.zshrc
```

**步骤 3：生成 local.properties（Gradle 定位 SDK）**
```bash
cat > app/local.properties <<EOF
sdk.dir=$HOME/Library/Android/sdk
EOF
```
> ⚠️ `local.properties` 含本机绝对路径，**必须加入 `.gitignore`，不可提交**。

**步骤 4：补全 Gradle Wrapper 的 jar（当前仓库只有 .properties）**
```bash
cd app
# 用本机临时 gradle 生成完整 wrapper，或从任一 KMP 模板拷贝 gradle-wrapper.jar
brew install gradle
gradle wrapper --gradle-version 8.9
# 之后即可用 ./gradlew，不再依赖全局 gradle
```

**步骤 5：编译 Android（第一次会拉依赖，耐心等）**
```bash
cd app
./gradlew :composeApp:assembleDebug --stacktrace
# 期望：BUILD SUCCESSFUL，产物在 composeApp/build/outputs/apk/debug/
```

**步骤 6：编译 iOS framework（需 Xcode 已装）**
```bash
sudo xcode-select -s /Applications/Xcode.app/Contents/Developer
cd app
./gradlew :composeApp:linkDebugFrameworkIosSimulatorArm64 --stacktrace
# 这一步会真正触发 AVFoundation cinterop 编译，是验证 R1 最高风险点的关键
```

### 实施注意事项（按踩坑概率排序）
1. **iOS cinterop 报错优先在此暴露**：`AudioCapture.ios.kt` 的 `convertToBuffer` 闭包签名、`int16ChannelData` 取值、`AVAudioConverterInputStatus` 常量，是 Kotlin/Native 绑定最易出错处。报错时对照 Xcode 中 `AVAudioConverter.h` 的实际签名调整，**不要猜**。
2. **Compose 编译器版本须与 Kotlin 对齐**：本项目用 Kotlin 2.1.0 + `composeCompiler` 插件同版本，已在 `libs.versions.toml` 锁定，勿单独升级其一。
3. **首次编译慢**：KMP 要下载 Kotlin/Native 工具链（约 1GB），失败多为网络问题，配置 Maven 镜像可缓解。
4. **磁盘**：本机剩余约 287GB 足够；Xcode + Android SDK + KMP 缓存合计约 30GB。

### CI 门禁（GitHub Actions）
新建 `.github/workflows/native-build.yml`：
```yaml
name: native-build
on:
  push:
    paths: ['app/**']
  pull_request:
    paths: ['app/**']

jobs:
  android:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: '17' }
      - uses: android-actions/setup-android@v3
      - name: Assemble Android debug
        working-directory: app
        run: ./gradlew :composeApp:assembleDebug --stacktrace

  ios-framework:
    runs-on: macos-14   # 自带 Xcode
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: '17' }
      - name: Link iOS framework (含 cinterop 编译)
        working-directory: app
        run: ./gradlew :composeApp:linkDebugFrameworkIosSimulatorArm64 --stacktrace
```

### 预期效果
- 本机两条编译命令均 `BUILD SUCCESSFUL`。
- CI 在每次改动 `app/**` 时自动编译三端，红灯即拦截合并。
- iOS cinterop 的真实正确性被编译期强制验证。

### 有效性验证
| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| Android 可编译 | `./gradlew :composeApp:assembleDebug` | 生成 APK |
| iOS 可编译 | `./gradlew :composeApp:linkDebugFrameworkIosSimulatorArm64` | 生成 framework |
| CI 门禁生效 | 故意改坏一行 Kotlin 推送 | CI 变红 |
| cinterop 正确 | iOS framework 链接成功 | 无 unresolved reference |

## R2｜实时音频体验无法离线验证

### 问题回顾
barge-in ≤300ms、端到端延迟 <1.2s 依赖真机麦克风/扬声器/AEC/网络，纯代码无法验证。`LocalVad` 阈值（RMS 0.04 / 4 帧）是从 Web 端搬来的未标定经验值。

### 目标
建立**可量化、可复现**的真机声学测试协议，把"感觉流畅"变成"打断 280ms / 延迟 950ms"这种带数字的结论。

### 工具/资源清单
| 工具 | 用途 |
|------|------|
| Android 真机 ×2（不同档位）+ iOS 真机 ×2 | 覆盖机型 AEC 差异 |
| 安静房间 + 可控外放音源 | 标定回采误触发 |
| 秒表/录屏 240fps（手机自带慢动作） | 测打断延迟 |
| 后端时间戳日志 | 测端到端延迟 |
| 测试台账（下方模板） | 量化记录 |

### 操作步骤

**步骤 1：给关键链路打时间戳（仅测试期，发布前移除或降级为 DEBUG）**

客户端 `CallController` 在三处埋点（伪代码，落地时用平台日志）：
```kotlin
// 1) 本地 VAD 判定打断的瞬间
private fun onBargeDetected() { log("BARGE_LOCAL t=${nowMs()}") }
// 2) 调用 player.clear() 后
private fun afterClear() { log("PLAYER_CLEARED t=${nowMs()}") }
// 3) 收到服务端 barge_in 兜底
// applyControl(BargeIn) 里 log("BARGE_SERVER t=${nowMs()}")
```
后端在 `session/manager.py` 的 `_handle_event`（已存在）记录服务端检测到用户说话的时间戳。

**步骤 2：测「打断延迟」（核心指标 ≤300ms）**
1. Agent 持续播报一段长 TTS（如念一段长文本）。
2. 测试者在固定时刻说"停"，用 240fps 慢动作录屏同时拍摄"说话口型"与"扬声器停播"。
3. 逐帧数：从声波起点到声音停止的帧数 × (1000/240) = 打断延迟 ms。
4. 每机型重复 10 次，取 P50/P95。

**步骤 3：测「端到端延迟」（≤1.2s）**
1. 测试者说一句短问句，记录说完瞬间 t0（录屏帧）。
2. 记录 Agent 首个音频帧到达扬声器出声 t1（录屏帧）。
3. 延迟 = t1 − t0。重复 10 次取 P50/P95。

**步骤 4：测「AEC / 自播误触发」（关键质量项）**
1. Agent 正常播报，测试者**全程不说话**。
2. 观察是否发生本地 VAD 误打断（player 被异常 clear）。
3. 在最大音量、外放、手机贴桌面震动等恶劣条件各测 10 次。
4. 误触发率必须 = 0%；若 >0，说明 AEC 未生效或 VAD 阈值过低。

**步骤 5：标定 VAD 阈值**
- 若 R2-步骤4 出现误触发 → 调高 `LocalVad` 的 `rmsThreshold`（0.04 → 0.05/0.06）或 `triggerFrames`（4 → 5/6）。
- 若真人说话打断不灵敏（延迟 >300ms 且漏触发）→ 调低阈值。
- **每改一次，回归步骤 2 + 步骤 4**，直到两者同时达标。把最终值连同机型记入台账。

### 测试台账模板
```markdown
| 机型 | 系统 | 打断P50 | 打断P95 | 端到端P50 | 端到端P95 | 误触发率 | VAD阈值 | 结论 |
|------|------|---------|---------|-----------|-----------|---------|---------|------|
| 小米14 | A14 | 270ms | 310ms | 920ms | 1150ms | 0% | 0.04/4 | PASS |
| iPhone13 | i17 | ... | ... | ... | ... | ... | ... | ... |
```

### 实施注意事项
1. **先解决 AEC 再调 VAD**：顺序反了会把 AEC 问题误判成阈值问题，无限返工。
2. **机型矩阵不可省**：低端 Android 的 AEC 质量参差，是误触发重灾区，必须纳入。
3. **网络变量隔离**：测打断（纯本地）和测端到端（含网络）分开，否则数字无法归因。
4. **时间戳日志发布前处理**：避免线上日志泄漏或性能损耗。

### 预期效果
- 每个目标机型都有一行带数字的 PASS/FAIL 结论。
- VAD 阈值从"拍脑袋"变成"按机型标定"。
- 误触发率量化为 0%。

### 有效性验证
- 打断 P95 ≤ 300ms、端到端 P95 ≤ 1.2s、误触发率 = 0%，三项全绿即 R2 关闭。
- 任一项不达标，台账记录具体数字 + 调整动作，进入下一轮。

## R3｜隐私硬边界缺持续校验

### 问题回顾
"子女端绝不落原始对话"散落在 distiller / signals 多处，仅靠一次性自测，无回归门禁。新增功能（mood_score、未来 TimesFM 摘要）任一回归都可能带出原文。

### 目标
把隐私约束固化为**自动化回归测试 + CI 强门禁**：任何改动只要让原文进入落库/外传路径，CI 立即变红。

### 工具/资源清单
| 工具 | 用途 |
|------|------|
| pytest | 测试框架（替代现有手写 assert 脚本，便于 CI 聚合） |
| 现有 `test_signals.py` 隐私断言 | 复用其"secret 不入库"思路 |
| CI（同 R1 的 Actions） | 门禁 |

### 操作步骤

**步骤 1：抽象统一的隐私断言工具**

新建 `backend/scripts/privacy_assert.py`：
```python
"""隐私硬边界统一断言：给定原始对话与任一对外产物，断言产物不含原文片段。"""
from __future__ import annotations

# 从原始对话中提取的"绝不允许外泄"的敏感片段（人名/地址/具体事件）
def assert_no_raw_leak(raw_sentences: list[str], outward_blob: str) -> None:
    """raw_sentences: 老人说的原话；outward_blob: 任何落库/推送给子女端的字符串。"""
    for s in raw_sentences:
        # 连续 ≥6 字的原文子串出现在产物中即判泄漏（避免单字误报）
        for i in range(len(s) - 5):
            frag = s[i:i + 6]
            if frag and frag in outward_blob:
                raise AssertionError(f"隐私泄漏：原文片段「{frag}」出现在对外产物中")
```

**步骤 2：覆盖全部对外路径的隐私回归测试**

新建 `backend/scripts/test_privacy.py`，对三条对外路径分别断言：
```python
"""隐私回归测试：覆盖 信号落库 / 信号摘要 / 记忆蒸馏 三条对外路径。
运行：.venv/bin/python -m backend.scripts.test_privacy
"""
from __future__ import annotations
import asyncio, tempfile
from pathlib import Path
from ..config import ArkConfig
from ..signals.engine import SignalService
from .privacy_assert import assert_no_raw_leak

SECRETS = ["我儿子叫张伟住朝阳区三号楼", "上周三在厨房摔倒了"]
RAW = [
    {"role": "user", "text": f"我有点难过，{SECRETS[0]}，他好久没来看我了。"},
    {"role": "user", "text": f"{SECRETS[1]}，到现在腰还疼。"},
]

def _ark_off() -> ArkConfig:
    cfg = ArkConfig.__new__(ArkConfig)
    object.__setattr__(cfg, "base_url", "x")
    object.__setattr__(cfg, "text_model", "x")
    return cfg

async def test_signal_path() -> None:
    tmp = Path(tempfile.gettempdir()) / "privacy_test.db"
    if tmp.exists(): tmp.unlink()
    svc = SignalService(str(tmp), _ark_off())  # 关闭方舟，走纯规则，离线可断言
    await svc.ensure_schema()
    await svc.generate("elder-x", RAW)
    items = await svc.list_signals("elder-x")
    assert_no_raw_leak(SECRETS, str(items))
    print("[PASS] 信号落库路径无原文泄漏")

async def main() -> None:
    await test_signal_path()
    # TODO: 接入 distiller 后补 test_memory_path / 接入 TimesFM 摘要后补 test_forecast_path
    print("\n隐私回归测试通过 ✅")

if __name__ == "__main__":
    asyncio.run(main())
```

**步骤 3：把隐私测试设为 CI 阻断项**

`.github/workflows/backend-test.yml`：
```yaml
name: backend-test
on:
  push: { paths: ['backend/**'] }
  pull_request: { paths: ['backend/**'] }
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - run: pip install -r backend/requirements.txt
      - name: 隐私回归（阻断项）
        run: python -m backend.scripts.test_privacy
      - name: 信号自测
        run: python -m backend.scripts.test_signals
```

**步骤 4：新增对外功能时的强制流程**
> 任何新增"会写库或推送给子女端"的功能（如 TimesFM 情绪摘要），**必须**在 `test_privacy.py` 增加对应路径断言，否则 PR 不予合并。把这条写入团队 CONTRIBUTING / Code Review 检查单。

### 实施注意事项
1. **测试必须能离线跑**：强制关闭方舟（`_ark_off()`），走规则降级路径断言，避免 CI 依赖外部 LLM。
2. **LLM 摘要路径要单独建采样测试**：方舟开启时，prompt 改动可能复述原话。建立一个"开方舟、固定输入、人工+自动双审"的采样集（≥20 条），定期跑。
3. **片段长度阈值**：6 字是经验值，过短误报、过长漏报，可按实际语料调。

### 预期效果
- 三条对外路径全部有自动断言。
- 任何让原文入库的回归在 CI 即被拦截。
- 新功能上线前隐私校验流程制度化。

### 有效性验证
- 故意在 `engine.py` 的 summary 里拼接一句原话 → `test_privacy` 必须失败。
- CI 中隐私测试为必过项，失败即阻断合并。

## R4｜外部语音服务单点无降级

### 问题回顾
"听+说"全双工走火山 Seeduplex（`engine/seeduplex.py`），记忆/信号走方舟 LLM。外部服务抖动/限流/鉴权变更会直接中断核心通话，且语音侧无降级路径。

### 目标
1. 启动时校验凭证，避免"跑起来才发现密钥错"。
2. 运行时探测外部服务健康，异常时给老人**友好语音兜底**而非静默卡死。
3. 方舟侧沿用已有"无 key 降级规则"范式，补齐语音侧的失败可观测。

### 工具/资源清单
| 工具 | 用途 |
|------|------|
| 现有 `/healthz` 端点 | 扩展为含依赖探测 |
| 现有 `engine/fake.py` | 作为语音链路的本地降级引擎 |
| 现有 `ArkConfig.enabled` 范式 | 复用降级思路 |

### 操作步骤

**步骤 1：启动期凭证校验（fail-fast）**

在 `server.py` 的 startup 钩子增加（不破坏现有惰性校验，仅在显式开启时检查）：
```python
@app.on_event("startup")
async def _verify_credentials() -> None:
    # 仅当使用 seeduplex 引擎时校验语音凭证，避免无关启动失败
    if cfg.engine == "seeduplex":
        try:
            _ = cfg.volc.app_id          # 触发 require()，缺失即抛
            _ = cfg.volc.access_token
            logger.info("火山语音凭证已就绪")
        except RuntimeError as e:
            logger.error("语音凭证缺失，将无法发起通话：%s", e)
```

**步骤 2：扩展健康探针**

把现有 `/healthz` 升级为分项健康（不实际调用外部，避免探针自身打爆配额，只报配置就绪态 + 最近一次会话结果）：
```python
@app.get("/healthz")
async def healthz() -> dict:
    return {
        "status": "ok",
        "engine": cfg.engine,
        "volc_ready": bool(cfg.volc.model_version),
        "ark_enabled": cfg.ark.enabled,   # 已有属性
        "model": cfg.volc.model_version,
    }
```

**步骤 3：语音链路失败的用户级兜底**

`server.py` 的 `elder_ws` 已有建连失败处理（发 `connect_failed`）。补一条：**当 seeduplex 建连失败时，可切换到 `fake` 引擎播一句预录兜底语音**"网络不太好，我们待会儿再聊好吗"，让老人不至于面对死寂。落点在 `create_engine` 失败的 except 分支：
```python
try:
    await session.start()
except Exception:
    logger.exception("会话启动失败，尝试降级兜底")
    await client_send('{"type":"status","status":"error","detail":"connect_failed"}')
    # 可选：用 fake 引擎播一段预录安抚音频后再关闭
    await ws.close()
    return
```
> 注：是否真正切 fake 引擎播兜底音，取决于产品是否接受"降级通话"。最低限度是明确的错误状态 + 前端友好提示（客户端 `CallScreen` 已有 ERROR 态文案）。

**步骤 4：外部依赖的运维约束清单（交付给运维）**
- 火山/方舟凭证用环境变量注入，**不入仓**（现状已合规，`.env` 在 `.gitignore`）。
- 记录两套鉴权体系互不通用（VOLC_* vs ARK_API_KEY），换 key 时分别更新。
- 监控外部服务的错误率/时延，设告警阈值。
- 提前确认配额上限与计费，避免上线峰值被限流。

### 实施注意事项
1. **探针不要反向打爆配额**：healthz 只报配置态，不在探针里真连火山。
2. **降级要"有声"**：老年用户对静默无反馈极敏感，降级时务必给语音/大字提示。
3. **凭证校验区分引擎**：用 `fake` 引擎本地联调时不应因缺火山凭证而启动失败。

### 预期效果
- 凭证缺失在启动日志立即暴露。
- healthz 可被监控系统拉取，外部异常可观测。
- 通话失败时老人得到友好反馈而非卡死。

### 有效性验证
| 场景 | 操作 | 期望 |
|------|------|------|
| 凭证缺失 | 清空 VOLC_ACCESS_TOKEN 启动 | 日志报"凭证缺失" |
| 健康可观测 | `curl localhost:8000/healthz` | 返回各依赖就绪态 |
| 失败兜底 | 断网发起通话 | 前端显示友好错误，无静默卡死 |

## R5｜技能缺口与协议契约漂移

### 问题回顾
CMP+KMP 需 Kotlin + iOS 原生能力，团队主体是 Python/Web；Web elder 与 KMP elder 双端并存，控制帧协议在 `后端 manager.py`、`Web app.js`、`Kotlin ControlFrame.kt` 三处手工维护，易漂移。

### 目标
1. 协议建立**单一真源**，三端从同一份定义派生或据其校验。
2. 用自动化契约测试防止漂移。
3. 给出技能缺口的最小可执行补齐路径。

### 工具/资源清单
| 工具 | 用途 |
|------|------|
| 一份 `protocol.md` 协议规范 | 单一真源（人读 + 测试依据） |
| pytest + Kotlin test | 三端契约一致性测试 |
| 现有三处实现 | 对齐目标 |

### 操作步骤

**步骤 1：建立协议单一真源**

新建 `docs/protocol.md`，把控制帧契约固化为表（从 `manager.py#L9-L12` 抽取）：
```markdown
## 适老端 <-> 后端 控制帧契约 v1
### 上行（客户端 → 后端）
| type | 字段 | 说明 |
|------|------|------|
| hangup | 无 | 挂断 |
（二进制帧 = PCM 16k/mono/16bit-LE）

### 下行（后端 → 客户端）
| type | 字段 | 说明 |
|------|------|------|
| barge_in | 无 | 服务端打断信号 |
| text | role, text | 字幕 |
| status | status, detail | connected/ended/error |
（二进制帧 = TTS PCM 24k/mono/16bit-LE）
```
> 任何协议变更**先改这份文档并评审**，再改三端实现。文档是裁判。

**步骤 2：后端契约快照测试**

新建 `backend/scripts/test_protocol_contract.py`，断言后端发出的帧 type 集合与文档一致：
```python
"""协议契约快照：后端实际产出的控制帧类型必须与 docs/protocol.md 声明一致。
运行：.venv/bin/python -m backend.scripts.test_protocol_contract
"""
# 文档声明的下行 type 全集（改协议时必须同步改这里 + docs/protocol.md）
DOWNSTREAM_TYPES = {"barge_in", "text", "status"}
UPSTREAM_TYPES = {"hangup"}

# 从 session/manager.py 静态扫描实际使用的 type 字面量做交叉校验
import re, pathlib
SRC = pathlib.Path(__file__).resolve().parent.parent / "session" / "manager.py"

def test_downstream_in_sync() -> None:
    text = SRC.read_text(encoding="utf-8")
    used = set(re.findall(r'"type":\s*"(\w+)"', text))
    # manager 里出现的 type 必须都在文档声明内（允许文档是超集）
    unknown = used - (DOWNSTREAM_TYPES | UPSTREAM_TYPES)
    assert not unknown, f"manager.py 出现文档未声明的帧类型：{unknown}"
    print("[PASS] 后端帧类型与协议文档一致")

if __name__ == "__main__":
    test_downstream_in_sync()
    print("\n协议契约测试通过 ✅")
```

**步骤 3：Kotlin 端契约单测**

在 `app/composeApp/src/commonTest/kotlin/.../ControlFrameTest.kt` 断言解析覆盖全部下行 type：
```kotlin
import com.elder.companion.protocol.ControlFrame
import com.elder.companion.protocol.ServerFrame
import kotlin.test.Test
import kotlin.test.assertTrue

class ControlFrameTest {
    @Test fun parses_all_downstream_types() {
        assertTrue(ControlFrame.parse("""{"type":"barge_in"}""") is ServerFrame.BargeIn)
        assertTrue(ControlFrame.parse("""{"type":"text","role":"user","text":"hi"}""") is ServerFrame.Text)
        assertTrue(ControlFrame.parse("""{"type":"status","status":"connected"}""") is ServerFrame.Status)
        assertTrue(ControlFrame.parse("""{"type":"garbage"}""") is ServerFrame.Unknown)
    }
    @Test fun hangup_shape_is_stable() {
        assertTrue(ControlFrame.hangup().contains("\"hangup\""))
    }
}
```
CI 跑 `./gradlew :composeApp:iosSimulatorArm64Test`（或 `testDebugUnitTest`）。

**步骤 4：技能缺口补齐路径（最小可执行）**
| 缺口 | 最小动作 | 负责人画像 |
|------|---------|-----------|
| Kotlin/KMP | 1 人完成官方 KMP 教程 + 跑通本项目 R1 编译 | 现有 Android 或后端工程师 |

### 实施注意事项
1. **协议改动走"文档先行"**：先改 `docs/protocol.md` 并评审，再改三端 + 同步测试常量，顺序不可乱。
2. **契约测试是静态扫描**：步骤 2 用正则扫 `manager.py`，重构挪动字面量时需确认正则仍命中（或改为运行时收集）。
3. **是否保留双端是成本决策**：双端并存意味着每次协议变更三处同改，若团队小，建议 KMP 达标后收敛。

### 预期效果
- 协议有唯一裁判文档，三端据其校验。
- 后端/客户端各有契约测试，漂移即红灯。
- 技能缺口有具体补齐人选与动作，而非空泛"需要学习"。

### 有效性验证
- 在 `manager.py` 新增一个文档未声明的 type → `test_protocol_contract` 失败。
- Kotlin `ControlFrameTest` 全过。
- 指定责任人完成 R1 本机编译，即视为 Kotlin 基础技能就位。
