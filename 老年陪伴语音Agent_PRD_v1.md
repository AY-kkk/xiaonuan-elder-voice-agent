## 1. 一句话概述（Executive Summary）

我们要为**独居/半独居老人**打造一款「老年陪伴语音 Agent」App：老人打开 App 即可与一个**使用子女克隆声音**的 AI 进行**全双工、可打断、低延迟的语音对话**；Agent 通过**产品层自建的分层记忆（重点事项 + 生活记忆）**记住老人的健康、习惯与情绪，并在子女端**只推送结构化摘要与关键信号**（不推送原始对话），从而在「陪伴老人」与「保护隐私、减轻子女焦虑」之间取得平衡。

本阶段（MVP）只做一件事并把它做透：**用火山豆包实时语音 API（Seeduplex）把端到端全双工语音链路稳定打通，并叠加最小可用的分层记忆与子女端信号摘要**。

---

## 2. 问题陈述（Problem Statement）

**谁有问题：** 60 岁以上、子女不在身边的独居或半独居老人，以及关心他们但无法时刻陪伴的成年子女。

**问题是什么：**
- 老人日常缺乏即时、自然的对话陪伴，孤独感强；现有智能音箱"一问一答、答非所问、必须喊唤醒词"，对老人不友好。
- 老人对打字、复杂 UI 几乎无操作能力，**语音是唯一自然入口**，但市面产品的语音交互延迟高、不能打断、不能连续聊。
- 子女想了解父母状态（吃没吃药、情绪好不好、有没有异常），但**没有低打扰的渠道**；直接看监控/聊天记录又涉及隐私。

**为什么痛：**
- 老人孤独是普遍且持续的刚需，陪伴类产品天花板高。
- "能对话"和"能像真人一样自然对话"之间存在巨大体验鸿沟——延迟、打断、记忆是三道硬门槛。
- 子女的"知情焦虑"与老人的"隐私尊严"天然冲突，需要产品化的中间层来调和。

**证据（待验证假设）：**
- `[assumption–待验证]` 老人对"用子女声音陪聊"有显著情感偏好，是核心差异化卖点。
- `[assumption–待验证]` 子女更愿意接受"摘要/信号"而非"全量对话"，既减负又避免侵犯父母隐私。

## 3. 目标用户与画像（Target Users & Personas）

### 3.1 主画像 A — 被陪伴的老人「张阿姨」（68 岁）
- **背景**：退休教师，独居，子女在外地。会用微信语音，但打字慢、怕操作错。眼睛花，看不清小字。
- **典型语录**：「我就想有个人说说话」「这字太小了我看不见，能不能直接跟它讲」。
- **痛点**：孤独；不会用复杂 App；记性变差，常忘记吃药、忘记今天星期几。
- **JTBD**：打开就能聊天、声音熟悉亲切、它能记得我说过的事、别让我学新东西。

### 3.2 主画像 B — 远程关心的子女「李先生」（40 岁）
- **背景**：一线城市工作，工作忙，每天能给父母打电话的时间有限，担心父母健康和情绪。
- **痛点**：信息缺失、知情焦虑；又不想监控父母、侵犯隐私。
- **JTBD**：低打扰地知道父母今天状态如何、有没有异常信号、需不需要我介入；同时给父母一个"我陪着你"的情感载体（克隆声音）。

### 3.3 次画像
- **声音被克隆的子女/家人**（可能就是 B，也可能是其他家庭成员）。
- 后续可扩展：社区/养老机构运营方（非 MVP 范围）。

## 4. 战略背景（Strategic Context）

- **业务目标**：验证"全双工语音陪伴 + 分层记忆 + 子女信号"这一产品形态的**情感粘性与技术可行性**，跑通最小闭环，为后续商业化（订阅 / 硬件捆绑 / 银发养老服务）打基础。
- **为什么是现在（Why now）**：端到端实时语音大模型（火山豆包 Seeduplex）已具备**全双工、可打断、亚秒级**能力，让"像真人一样自然对话"第一次在工程上变得可行；声音克隆技术成熟，"子女声音陪伴"具备落地条件。
- **竞争格局**：传统智能音箱（唤醒词 + 一问一答，体验差）；通用聊天 App（无语音原生、无适老化、无家庭信号）。我们的差异点 = **全双工自然语音 + 子女克隆音 + 分层记忆 + 子女端隐私友好信号**四者组合。

## 5. 方案概述（Solution Overview）

### 5.1 产品形态
- **老年端 App**：极简界面，一个大的"通话/挂断"状态，打开即连，语音为唯一交互；视觉只做兜底（大字、状态指示）。
- **子女端 App / H5**：声音克隆录制入口、重点事项配置、每日/异常**信号摘要**查看。
- **后端服务**：作为唯一持有火山 API Key 的代理层，负责 WebSocket 转发、会话管理、记忆蒸馏与存储、信号生成与推送。

### 5.2 核心用户流程

**流程 1：老人语音陪聊（核心闭环）**
```
老人打开 App
  → 客户端与"自有后端"建立 WebSocket（不直连火山）
  → 后端持 API Key 与火山 Seeduplex 建立 WS（StartConnection → StartSession）
  → 老人说话：客户端采集 PCM(16k/mono/16bit) → 后端透传给火山
  → 火山全双工返回：音频流(TTS, 子女克隆音色) + 文本流 → 后端 → 客户端播放
  → 老人中途插话：触发 barge-in，客户端立即停播 + 后端同步打断信号
  → 老人挂断/超时 → FinishSession → 触发"会话结束"事件
```

**流程 2：记忆蒸馏（异步，与语音链路解耦）**
```
会话结束事件
  → 后端取本轮完整对话文本
  → 异步调用独立方舟 LLM 做蒸馏
  → 写入层级 A（重点事项，结构化）/ 层级 B（生活记忆）
  → 下次会话开始时，作为系统上下文注入 StartSession
```

**流程 3：子女端信号**
```
蒸馏结果 + 规则引擎（关键词/情绪/异常检测）
  → 生成"信号/摘要"（如：今日情绪正常、提到膝盖疼 2 次、未确认吃药）
  → 推送到子女端（只推摘要/信号，不推原始对话）
```

### 5.3 关键特性清单（MVP）
1. 全双工实时语音对话（基于火山 Seeduplex，可打断、连续聊）。
2. 子女声音克隆音色作为 Agent 输出音色。
3. 分层记忆：层级 A（重点事项结构化）+ 层级 B（生活记忆自动蒸馏）。
4. 会话结束后异步记忆蒸馏（独立方舟 LLM，与语音链路解耦）。
5. 子女端信号摘要（隐私友好，不含原始对话）。
6. 适老化老年端：打开即连、语音为主、大字兜底。

## 6. 本阶段做什么 / 不做什么（MVP Scope）

### 6.1 本阶段做什么（In Scope）
| 模块 | MVP 范围 |
|---|---|
| 语音链路 | 跑通"客户端 ↔ 后端 ↔ 火山 Seeduplex"全双工链路；支持打断；目标总体延迟 < 1.2s |
| 方言/音色 | **普通话**（子女克隆音，主路径）；**四川话 / 粤语**（目标做克隆，失败则降级到预置音色） |
| VAD | 老年端静音判停阈值 **800–1200ms**（比通用场景更宽容，照顾老人语速慢、停顿多） |
| 记忆 | 层级 A 重点事项（结构化，子女可在子女端预设）+ 层级 B 生活记忆（会话后自动蒸馏） |
| 蒸馏 | 会话结束异步调用独立方舟 LLM；与语音 WS 链路完全解耦 |
| 子女端 | 声音克隆录制 + 重点事项配置 + 每日/异常信号摘要查看 |
| 安全 | API Key 只存后端，客户端永不接触；客户端只连自有后端 |

### 6.2 本阶段不做（Out of Scope，附理由）
- ❌ **多模态（摄像头/视觉/屏幕理解）** —— MVP 聚焦语音陪伴闭环，视觉非核心价值，且增加适老化与隐私复杂度。
- ❌ **设备控制 / 执行类 Agent 能力（如帮老人操作其他 App）** —— 超出陪伴定位，风险高。
- ❌ **多老人/多家庭组织管理、机构版** —— 先验证单家庭闭环。
- ❌ **主动外呼 / 主动唤起老人** —— MVP 先做"老人主动打开"，主动关怀放二期。
- ❌ **超过基础三种方言的更多方言**、复杂情绪治疗/医疗诊断 —— 合规与质量风险，明确不做。
- ❌ **子女端查看原始对话** —— 与隐私原则冲突，明确不提供（核心产品决策）。

## 7. 技术架构（Technical Architecture）

### 7.1 总体架构分层

```
│  打断处理、极简 UI         │        │  信号摘要查看              │
└─────────────┬────────────┘        └─────────────┬────────────┘
              │ WSS（自有协议，仅连自有后端）          │ HTTPS
              ▼                                     ▼
┌─────────────────────────────────────────────────────────────┐

│  ① 语音 WS 代理转发（持有火山 API Key）                          │
│  ② 会话管理（session/dialog 生命周期、上下文注入）               │

│  ④ 蒸馏服务（异步队列 → 独立方舟 LLM）                          │

└───────┬───────────────────────────────┬─────────────────────┘

        ▼                                 ▼
┌─────────────────────────┐    ┌─────────────────────────┐
│ 火山豆包实时语音 Seeduplex │    │  火山方舟 LLM（蒸馏/信号）  │
│ wss://openspeech...s2s    │    │  （独立调用，异步解耦）     │
└─────────────────────────┘    └─────────────────────────┘
```

### 7.2 火山豆包实时语音（Seeduplex）接入要点
> 依据火山引擎官方 WS 协议文档（doc 6561/1594356）整理，落地前以官方最新文档为准。

**连接与鉴权**
- Endpoint：`wss://openspeech.bytedance.com/api/v3/s2s`
- 鉴权通过 HTTP Header 传递，**全部在后端完成，客户端永不接触**：
  - `X-Api-Resource-Id`：`volc.speech.dialog`（全双工端到端模型服务）
  - `X-Api-Access-Key`：控制台 Access Token
  - `X-Api-App-Key`：AppID / AppKey
  - `X-Api-Connect-Id`：客户端/后端生成的 UUID，用于全链路追踪

**二进制协议帧结构（v3）**：`Header(4B) + Payload Size(4B, 大端) + Payload`
- Byte0：协议版本（高4位）+ Header 长度（低4位），常见 `0x11`
- Byte1：Message Type（高4位）+ Flags（低4位）。Message Type：`0x1` FullRequest、`0x2` AudioOnly、`0x3` Event/Control、`0xF` Error；Flag `0x4` 表示携带 Event（JSON 元数据）
- Byte2：序列化方式（高4位，`0x1`=JSON）+ 压缩方式（低4位，`0x0` 不压缩 / `0x1` Gzip）
- Byte3：保留

**事件时序**
1. `StartConnection`（MsgType 1, Flag 4）→ 服务端 `ConnectionStarted`
2. `StartSession`：JSON Payload 配置 dialog（bot 信息、音色、系统上下文/记忆注入）
3. 持续发送 `AudioOnly`(MsgType 2) 流式音频帧
4. 服务端异步回传 `AudioResponse`（TTS 音频流）+ `TextResponse`（文本）
5. `FinishSession`（结束本轮、连接可复用）/ `FinishConnection`（彻底断开）

**音频格式**
- 原始 PCM、16000Hz、单声道、16bit、小端序
- 建议 20ms 一包（每包约 640 字节），平衡实时性与网络开销

**全双工 / 打断（barge-in）**
- 模型支持"边发边收"，机器人说话时老人可直接插话
- 服务端检测到用户语音会下发打断信号（如 StopSpeaking / ClearBuffer 类事件）；客户端收到后**立即清空本地播放缓冲并停播**，后端同步重置对话状态

**上下文限制（核心约束）**
- 语音模型自身上下文短（约 20 轮），**不能依赖它做长期记忆**——这正是我们必须在产品层自建分层 Memory 的根本原因（见 7.4）。

### 7.3 工程参考：X-OmniClaw 可借鉴模式
> 参考开源项目 [X-OmniClaw](https://github.com/Master-Frank/X-OmniClaw)（端侧多模态语音 Agent），抽象出**与本项目相关的可复用工程模式**（非照搬其多模态/设备执行能力，那些在我们 Out of Scope）：

| 可借鉴模式 | X-OmniClaw 做法 | 本项目落地方式 |
|---|---|---|
| **WebSocket 网关代理** | 自建网关层屏蔽云端模型直连，统一鉴权 | 后端作为唯一持 Key 的代理层，端→云→端透传火山二进制流，客户端永不直连火山 |
| **回声消除（AEC）** | observer 层接 WebRTC AEC，防止自播声被回采 | 客户端集成 AEC，避免 Agent 自己的 TTS 被当成老人说话误触发打断 |
| **打断处理** | TTS 管理器提供 stop()，监听新一轮录音即清空播放队列 | 收到 barge-in 即停播 + 清缓冲；与火山下发的打断信号双向对齐 |
| **分层记忆** | 会话级工作记忆（ArrayDeque）+ 长期 Profile | 层级 B 生活记忆（滚动短期）+ 层级 A 重点事项（长期结构化 Profile） |
| **API 适配器** | ApiAdapter 统一对接多家模型（含豆包）的 Header/鉴权 | 后端用适配器封装"实时语音 WS"与"方舟蒸馏 LLM"两条不同链路的鉴权 |

> 注：X-OmniClaw 为 Android 端侧架构，其"手机即网关"的形态我们改为**独立后端网关**（更利于 Key 安全与多端统一），其音频预处理/打断/分层记忆的工程思路可直接借鉴。

### 7.4 分层记忆设计（产品层自建）

| 维度 | 层级 A：重点事项 | 层级 B：生活记忆 |
|---|---|---|
| 内容 | 吃药提醒、慢病、忌口、重要日期、紧急联系人等 | 日常聊天中的偏好、近况、情绪、爱好、家长里短 |
| 结构 | 结构化（key-value / schema），可被子女在子女端预设/编辑 | 半结构化文本，自动蒸馏生成 |
| 来源 | 子女配置 + 对话中识别 | 会话结束后 LLM 自动蒸馏 |
| 注入 | 每次 StartSession 高优先级注入系统上下文 | 按相关度/时间衰减选择性注入 |
| 更新 | 实时/子女手动 | 异步蒸馏，会话后更新 |

**蒸馏链路（解耦设计）**：会话结束 → 写入异步队列 → 独立方舟 LLM 蒸馏 → 落库。**蒸馏完全不阻塞语音 WS 链路**，即使蒸馏失败也不影响下一次对话（降级为仅用已有记忆）。

### 7.5 数据流与隐私边界
- 原始对话音频/文本：仅在后端用于实时交互与蒸馏，**不向子女端开放**。
- 子女端可见：层级 A 重点事项（自己配置的）+ 信号摘要（蒸馏后的结论性信息）。
- API Key：仅后端环境变量/密钥管理，客户端与子女端均不可见。

## 8. 端到端链路打通保障（核心交付目标）

> 本节回答"如何保证端到端链路打通"——这是 MVP 的首要验收维度。

### 8.1 分段验证策略（自底向上打通）
| 阶段 | 验证目标 | 通过标准 |
|---|---|---|
| L0 后端↔火山 | 后端能建连、StartSession、收发音频 | 用测试音频跑通一轮完整问答，收到 TTS 音频流 |
| L1 客户端↔后端 | 自有 WS 协议透传 PCM、回传音频 | 端到端听到回复，无明显丢包 |
| L2 全双工/打断 | barge-in 生效 | 机器人说话时插话能在 ≤300ms 内停播并响应新话 |
| L3 记忆闭环 | 蒸馏→注入生效 | 第二次会话能引用第一次提到的事项 |
| L4 子女端信号 | 摘要正确生成并推送 | 异常关键词能触发信号，且不含原始对话 |

### 8.2 链路稳定性措施
- **连接管理**：心跳保活（默认无心跳约 60s 超时）、断线自动重连、会话级 `dialog_id` 续接。
- **音频时序**：固定 20ms 分包、上行节流、下行 jitter buffer，保证播放平滑。
- **打断双向对齐**：客户端本地 VAD 触发停播 + 服务端打断信号兜底，二者任一生效即停播，避免"抢话"。
- **降级策略**：
  - 克隆音色不可用 → 降级预置音色（四川话/粤语主用此降级路径）。
  - 蒸馏服务异常 → 跳过本轮蒸馏，下次仍用已有记忆，不影响通话。
  - 火山链路抖动 → 重连 + 友好语音提示（"我没太听清，你再说一遍好吗"）。
- **可观测性**：全链路 `X-Api-Connect-Id` 透传打点；埋点采集 TTFT、端到端延迟、打断成功率、重连率、蒸馏成功率。

### 8.3 性能目标（量化）
| 指标 | 目标 | 说明 |
|---|---|---|
| 端到端总体延迟 | **< 1.2s** | 老人说完 → 听到回复开头 |
| 首 Token / 首音频延迟（TTFT） | < 1s 为优 | 参考既有 SLA 标准 |
| 打断响应延迟 | ≤ 300ms | 插话到停播 |
| VAD 静音判停 | 800–1200ms | 老年端专用阈值 |
| 链路可用性 | ≥ 99.9%（MVP 目标） | 后续向 99.95%+ 演进 |

## 10. 用户故事与验收标准（User Stories & Acceptance Criteria）

**Epic 假设**：若我们为独居老人提供"用子女声音、可自然打断、能记住事情"的语音陪伴，并给子女隐私友好的信号，则能在 4 周内让试点老人日均对话 ≥10 轮、周留存 ≥50%。

**US-1 全双工对话**
> 作为张阿姨，我想打开 App 就能直接说话聊天，这样我不用学任何操作就有人陪我说话。
- Given 老人已打开 App 且网络正常
- When 老人开口说话并停顿 800–1200ms
- Then 在 1.2s 内听到用子女声音的自然回复，且能连续多轮对话

**US-2 打断（barge-in）**
> 作为张阿姨，我想在它还在说话时直接插话，这样对话像和真人聊天一样自然。
- Given Agent 正在播放回复
- When 老人开口插话
- Then 客户端 ≤300ms 内停播并响应新内容，不会"抢话/串话"

**US-3 重点事项记忆（层级 A）**
> 作为李先生，我想预设父母的吃药/慢病等重点事项，这样 Agent 能主动关照、我也放心。
- Given 子女在子女端配置了"每天提醒吃降压药"
- When 老人当天对话
- Then Agent 能在合适时机自然提及该事项；该事项跨会话稳定保留

**US-4 生活记忆蒸馏（层级 B）**
> 作为张阿姨，我想它记得我昨天说过的事，这样我感觉它真的在乎我。
- Given 上一次会话中老人提到"膝盖疼"
- When 下一次会话开始
- Then Agent 能自然关心"膝盖好点了吗"，且蒸馏过程未阻塞任何一次通话

**US-5 子女端信号（隐私友好）**
> 作为李先生，我想低打扰地知道父母今天状态，但不想偷看他们的聊天，这样既安心又尊重父母。
- Given 当日已发生对话
- When 子女打开子女端
- Then 看到结构化摘要/信号（情绪、关键提及、异常提醒），且**绝不出现任何原始对话内容**

**US-6 方言与音色降级**
> 作为说四川话的老人，我想它能用我熟悉的乡音或熟悉的声音陪我聊。
- Given 目标方言为四川话/粤语
- When 克隆音色可用 → 使用克隆音色；不可用 → 自动降级预置音色
- Then 对话可正常进行，不因音色问题中断

## 11. 依赖与风险（Dependencies & Risks）

| 类型 | 项 | 风险 | 缓解 |
|---|---|---|---|
| 技术依赖 | 火山豆包 Seeduplex WS API | 协议/配额/限流变化 | 以官方文档为准；申请足额配额；做重连与降级 |
| 技术依赖 | 火山方舟 LLM（蒸馏） | 蒸馏质量/稳定性 | 解耦异步、失败不阻塞、可重试 |
| 技术依赖 | 声音克隆能力 | 方言克隆质量不达标 | 普通话主路径优先；方言降级预置音色 |
| 体验风险 | 老人 AEC/环境噪声导致误打断 | 打断体验差 | 集成 AEC + 双向打断对齐 + 调参 VAD 阈值 |
| 隐私风险 | 原始对话泄露至子女端 | 信任崩塌、合规问题 | 架构层硬隔离，子女端只取摘要；安全评审 |
| 合规风险 | 老人健康/医疗类内容 | 误导风险 | 明确不做医疗诊断；敏感内容引导就医 |
| 产品风险 | "子女克隆音"情感接受度未知 | 核心卖点不成立 | 试点期 A/B（克隆音 vs 中性音）验证 |
