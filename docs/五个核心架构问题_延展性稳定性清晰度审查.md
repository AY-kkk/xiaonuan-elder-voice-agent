# 五个核心架构问题：延展性、稳定性与架构清晰度审查

> 审查口径：按“一个可以长期演进、可排障、可扩家庭/多角色/多端”的生产系统标准看。  
> 结论先说：当前项目已经能跑通闭环，但架构还停留在“功能堆起来了”的阶段。继续加能力前，先把下面五个问题处理掉，否则后面每个新需求都会变成补丁。

---

## 1. 隐私边界已经自相矛盾，这是最危险的架构信号

**严重级别：Critical**

`backend/api/elder.py` 明确写着：`life_memories` 是老人看自己的数据，**绝不进子女端路由**。但 `backend/api/parent.py` 的 `/api/parent/{elder_id}/memories` 又直接返回了 `life_memories`。

证据：

- `backend/api/elder.py:1-5`：声明 `life_memories` 不进子女端。
- `backend/api/parent.py:109-115`：子女端接口返回 `life_memories`。

这不是注释过期那么简单。它说明团队还没真正定义清楚“老人可见数据”和“家人可见数据”的边界。

**为什么这会毁掉延展性：**

后续一旦接入真实家庭数据、多长辈、多子女、授权分级，所有接口都要重新审。因为现在的模型没有回答一个基本问题：哪些数据属于老人本人，哪些数据可以被家人处理，哪些数据只能变成摘要信号。

**为什么这会毁掉稳定性：**

测试可能通过，但产品边界是错的。隐私问题不是运行时报错，它会在真实用户数据进入系统后才爆。

**建议修法：**

1. 把数据分成三类：`elder_private`、`family_reviewable`、`family_signal`。
2. 子女端 `/memories` 只返回 `family_reviewable`，不得直接返回 `life_memories`。
3. 在测试里加硬断言：父端响应结构不允许出现 `life_memories` 字段。
4. 删除或修正所有与实现冲突的注释，别让注释继续骗人。

---

## 2. 应用启动和依赖注入还是“全局变量拼装”，规模一大就会失控

**严重级别：Major**

后端在模块 import 时直接加载配置、创建服务实例、绑定路由全局变量。`parent_api`、`elder_api`、`character_api` 都靠模块级 `_store` / `_signals` / `_usage` / `_character` 保存依赖。

证据：

- `backend/server.py:32-60`：import 时创建 `cfg`、`app`、store、service 并 bind。
- `backend/api/parent.py:22-42`：模块级全局变量 + `bind()`。

这套写法适合演示，不适合长期系统。它把依赖关系藏进了 import 顺序。

**为什么这会毁掉延展性：**

你以后想做多环境、多租户、多 App 实例、按测试构造不同依赖，都会碰到同一个问题：服务不是显式传入的，是模块偷偷拿的。

**为什么这会毁掉稳定性：**

测试现在必须在 import `server` 前 monkeypatch 配置。这个约束没人会一直记得。以后一个测试文件 import 顺序不对，就会污染真实 DB 或拿错配置。

**建议修法：**

1. 引入 `create_app(config)`，禁止 import 时创建真实服务。
2. 定义 `AppContainer`，集中持有 `MemoryStore`、`SignalService`、`CharacterService`、`UsageStore`。
3. 路由通过 FastAPI 的依赖注入（DI，Dependency Injection）拿服务，不再用模块级全局变量。
4. 测试只构造测试容器，不 monkeypatch `load_config()`。

---

## 3. 角色生成有两套路由在写同一个服务，架构边界不清楚

**严重级别：Major**

现在角色能力同时暴露在：

- 子女端语义：`/api/parent/{elder_id}/characters...`
- 旧接口语义：`/api/character/{elder_id}...`

证据：

- `backend/server.py:24-60`：同时 include `parent_api` 和 `character_api`。
- `backend/api/parent.py:189-267`：父端角色生成完整写路径。
- `backend/api/character.py:52-134`：旧角色接口仍然保留完整写路径。

这不是兼容性，这是两个入口在同时拥有同一个业务能力。

**为什么这会毁掉架构清晰度：**

SPEC 已经说“子女端成为唯一角色生成与配置入口”，但代码没有执行这个边界。以后谁要加鉴权、审计、授权确认、限流，会问：到底拦哪条路由？

**为什么这会毁掉稳定性：**

两套路由的校验和文案会逐渐漂移。一个接口修了授权确认，另一个忘了；一个接口限制音频大小，另一个限制不一致。事故就是这么来的。

**建议修法：**

1. `/api/character` 立刻标记 deprecated，并在文档里写下下线时间。
2. 如果必须兼容，只允许它转调父端 service 层的同一个 use case，不保留独立校验逻辑。
3. 对外 API（应用程序编程接口）只保留 `/api/parent/{elder_id}/characters...`。
4. 增加路由契约测试：前端不得调用 `/api/character`。

## 4. 角色状态机是字符串到处飞，没有形成真正的状态模型

**严重级别：Major**

角色状态散落在数据库字段、Python 常量、服务判断和前端文案里。状态本身没有成为一个明确的领域模型。

证据：

- `backend/character/store.py:20-22`：状态只是字符串 tuple。
- `backend/character/store.py:196-224`：`sync_status` 由声音和说话方式状态临时推导。
- `web/parent/shared.js:72-87`：前端重新映射一遍状态文案。

当前能跑，是因为路径还少。等增加“失败重试、撤回同步、重新训练、授权过期、多版本声音”时，这套状态会开始互相踩。

**为什么这会毁掉延展性：**

新状态只能继续加字符串和 if 判断，没有地方能回答“从 A 状态能不能转到 B 状态”。

**为什么这会毁掉稳定性：**

数据库没有 `CHECK` 约束，`is_active` 也没有从数据库层保证每个 elder 只有一个 active。现在靠应用代码自觉，一旦出现并发、脚本修库、旧接口写入，就可能出现非法状态。

**建议修法：**

1. 定义 `CharacterStateMachine`，集中声明允许的状态和转换。
2. 数据库加 `CHECK` 约束，至少约束 `voice_status`、`persona_status`、`sync_status`。
3. 数据库加局部唯一约束：同一个 `elder_id` 只能有一个 `is_active=1`。
4. 把前端状态文案从后端返回，或至少由后端返回 `human_status`，避免前后端各翻译一遍。

## 5. 异步任务没有可观测闭环，失败会静默丢失

**严重级别：Major**

会话结束后，记忆蒸馏和信号生成通过 `asyncio.create_task()` 直接丢到后台，没有任务登记、没有失败落库、没有重试、没有超时策略、没有告警。

证据：

- `backend/server.py:90-96`：会话结束直接 create task。
- `backend/server.py:155-169`：`session.run_until_end()` 被 cancel，但没有 await cancellation 完成。

这是典型的“看起来异步解耦，实际上丢任务没人知道”。

**为什么这会毁掉稳定性：**

用户通话结束后，前端以为处理完成；后台任务失败了，信号没有生成、记忆没有落库。没有错误返回，也没有运维入口。系统会表现为“偶现少数据”，这是最难排的故障类型。

**为什么这会毁掉架构清晰度：**

现在没有“后台任务层”这个概念。任务是业务逻辑的一部分，却散在 gateway 里。gateway 应该接入会话，不应该顺手承担任务调度。

**建议修法：**

1. 引入 `BackgroundJobService`，至少记录：任务类型、elder_id、状态、错误、开始/结束时间。
2. `create_task()` 包一层统一 runner，捕获异常并结构化日志。
3. 关键任务增加重试策略和最大重试次数。
4. `recv_task.cancel()` 后 await 并吞掉 `CancelledError`，确保资源真的释放。
5. `/healthz` 不只报凭证状态，还要报最近后台任务失败数。

---

## 最刻薄但必要的总结

当前项目最大的问题不是“功能不够”，而是很多边界只存在于文档和注释里，代码没有强制它们成立。

优先级应该是：

1. 先修隐私边界。
2. 再清掉双 API 写路径。
3. 然后把全局依赖改成显式应用工厂。
4. 接着收敛角色状态机。
5. 最后补后台任务可观测性。

这五件事不做，继续加页面、加角色、加家庭关系，只会把系统从“能跑的原型”推向“没人敢改的原型”。
