# 小暖人文体验与端侧职责重构方案

> 目标：把当前“技术功能集合”重构为一个更有温度、更符合家庭照护分工的老年语音 Agent。老人端只保留轻量陪伴与选择，复杂配置交给子女端完成。

---

## 1. 当前核心问题

### 1.1 缺少人文气息

当前产品已经具备实时语音、记忆蒸馏、信号提炼、声音克隆等能力，但界面和流程仍偏“工程演示”：

- 页面强调“通话、成本、克隆、配置”，不像一个照护型陪伴产品。
- 老人端文案偏功能说明，缺少“被惦记、被照顾、被家人准备好”的情绪感。
- 角色再生被呈现为“换声音和性子”，像技术配置，不像“家人把一个熟悉的人带到你身边”。
- 记忆被当作 AI 的数据能力，而不是家庭共同维护的“关照线索”。

### 1.2 老人端承担了过多配置责任

老人端目前包含角色创建、音色克隆、人格语料上传等流程。这不合理：

- 老人不应该理解 speaker_id、音频格式、训练状态、人格语料。
- 老人也不适合处理录音上传、隐私授权、角色配置失败重试等复杂任务。
- 对老人而言，“换个声音”应该只是选择谁来陪我，而不是创建和训练角色。

### 1.3 子女端职责不足

子女端目前主要查看信号、管理重点事项和成本。它应该升级为家庭照护工作台：

- 为老人准备角色。
- 上传授权音频。
- 整理角色说话风格。
- 管理老人重要记忆。
- 审核 AI 从对话中提炼出的事项。
- 给老人端同步“可用角色”和“温柔提示”。

### 1.4 “成本”命名不适合家庭照护场景

“成本”是开发/运营视角，不是用户视角。对子女来说，更自然的是：

- 充值
- 余额
- 本月使用
- 语音陪伴时长
- 关怀分析次数
- 计费说明

因此子女端底部导航中的“成本”应改为“充值”，页面内容从“花了多少钱”改为“余额与计费记录”。

## 2. 产品重新定位

### 2.1 一句话定位

小暖不是“给老人使用的 AI 工具”，而是“子女提前准备好、老人一键享受的家庭语音陪伴”。

### 2.2 两端职责

| 端 | 新定位 | 应该做什么 | 不应该做什么 |
|---|---|---|---|
| 老人端 | 陪伴端 | 一键说话、看见家人的关心、选择已准备好的陪伴角色 | 创建角色、上传音频、填写语料、处理训练失败 |
| 子女端 | 照护配置端 | 准备角色、上传音色、维护记忆、充值、查看信号 | 展示原始对话、替老人做医疗判断 |

### 2.3 体验原则

1. 老人端只出现“我能理解的东西”：谁在陪我、今天要不要聊、家人有没有给我准备新声音。
2. 子女端承担“技术和配置”：音色、角色、人设、记忆、充值、计费。
3. 所有 AI 能力都要转译成家庭语言，而不是技术语言。
4. 任何关照都要克制，不制造监控感。

## 3. 老人端重构方案

### 3.1 移除老人端复杂配置

从老人端移除或隐藏：

- 新建角色
- 上传录音
- 填写 speaker_id
- 粘贴人格语料
- 查看训练状态细节

原页面 `web/elder/character.html` 不再作为老人端配置页，而应重构为“角色选择页”。

### 3.2 底部导航调整

当前：

```text
通话 / 聊过的事 / 换个声音 / 切换身份
```

建议：

```text
通话 / 关心我的事 / 选择陪伴 / 切换身份
```

其中“选择陪伴”只展示子女端已经准备好的角色。

### 3.3 老人端角色选择页

老人看到的是：

- 默认小暖
- 女儿小芳
- 老伴阿琴
- 老朋友老李

每个角色卡片只展示老人能理解的信息：

```text
女儿小芳
她的声音已经准备好了
用她陪我说话
```

如果角色尚未完全就绪：

```text
女儿小芳
家人正在准备她的声音，准备好后会告诉你
```

老人端不展示：

- 训练中
- speaker_id
- 人格蒸馏
- 音频格式
- token
- API

### 3.4 角色同步提示

当子女端准备好并启用角色后，老人端下一次进入首页时出现温柔提示：

```text
小芳给你准备了一个熟悉的声音。
今天想让她陪你说说话吗？
```

按钮：

```text
让小芳陪我
还是小暖陪我
```

如果老人不选择，默认不强制切换，避免惊吓。

### 3.5 通话首页增加人文层

老人端首页从“轻触和家人说说话”升级为更有陪伴感的动态文案：

| 场景 | 文案 |
|---|---|
| 早晨 | 早上好，今天也慢慢来 |
| 下午 | 下午好，想不想聊会儿天 |
| 晚上 | 晚上好，我在这儿陪你说说话 |
| 子女刚配置角色 | 小芳给你准备了熟悉的声音 |
| 有重点事项 | 今天记得按时吃药，慢慢来，不着急 |

### 3.6 老人端接口需求

新增老人端轻量角色接口：

```http
GET /api/elder/{elder_id}/companions
```

返回：

```json
{
  "active_character_id": 3,
  "items": [
    {
      "id": 0,
      "name": "小暖",
      "relation": "默认陪伴",
      "ready": true,
      "is_active": false,
      "elder_copy": "我一直在，想聊就点我"
    },
    {
      "id": 3,
      "name": "女儿小芳",
      "relation": "女儿",
      "ready": true,
      "is_active": true,
      "elder_copy": "小芳的声音已经准备好了"
    }
  ],
  "notice": {
    "type": "new_character_ready",
    "text": "小芳给你准备了一个熟悉的声音，想让她陪你说说话吗？",
    "character_id": 3
  }
}
```

老人端只调用：

```http
POST /api/elder/{elder_id}/companions/{character_id}/activate
```

## 4. 子女端重构方案

### 4.1 子女端新增“陪伴角色”模块

子女端负责完整角色创建流程：

```text
创建角色 → 上传声音 → 填写说话风格/记忆 → 预览 → 启用并同步给老人
```

建议子女端导航：

```text
信号 / 事项 / 角色 / 充值 / 我的
```

如果底部导航空间不足，可以把“事项”和“记忆审核”合并为“照护”。

### 4.2 子女端角色创建流程

#### 第一步：这个人是谁

字段：

- 角色名字：女儿小芳
- 与老人关系：女儿
- 老人习惯怎么称呼 TA：小芳 / 闺女 / 阿芳

文案：

```text
先告诉小暖，这个声音对老人意味着谁。
```

#### 第二步：上传授权声音

字段：

- 上传录音
- 朗读文本
- 授权确认 checkbox

授权文案：

```text
我确认已获得该声音本人或合法代理人的授权，仅用于陪伴本家庭老人。
```

这一步由子女完成，不出现在老人端。

#### 第三步：补充说话方式

字段：

- TA 平时怎么称呼老人
- TA 说话是温柔、爽朗、慢条斯理，还是爱开玩笑
- 可粘贴几段聊天文字
- 不希望 TA 说的话

文案不要叫“人格蒸馏”，应叫：

```text
让小暖学一学 TA 的说话方式
```

#### 第四步：补充关照记忆

子女可以填：

- 老人喜欢聊什么
- 最近需要多关心什么
- 哪些事不要主动提
- 用药、慢病、忌口、重要日期

这里和已有 `key_facts`、`life_memories` 打通。

#### 第五步：同步给老人

当音色与人设都 ready 后，子女点击：

```text
同步给老人端
```

同步后老人端出现提示，而不是立即强制切换。

### 4.3 子女端角色状态

角色状态建议分为：

| 状态 | 含义 | 子女端文案 | 老人端表现 |
|---|---|---|---|
| draft | 未完成 | 还差几步 | 不展示 |
| voice_training | 声音训练中 | 正在准备声音 | 可不展示 |
| persona_ready | 说话方式已准备 | 还未同步 | 不展示 |
| ready | 可启用 | 可以同步给老人 | 可展示 |
| synced | 已同步 | 老人端已可选择 | 展示 |
| active | 当前使用 | 正在陪伴老人 | 默认选中 |
| failed | 失败 | 需要重新上传 | 不展示 |

## 5. 记忆上传与审核重构

### 5.1 老人端不再管理记忆

老人端可以保留“关心我的事”，但只用于温柔展示：

```text
小暖记得你喜欢下午听评书。
小暖记得你最近膝盖不太舒服，会提醒你慢慢走。
```

不提供复杂编辑。

### 5.2 子女端管理记忆

子女端新增“照护记忆”能力：

- 添加重点事项
- 添加生活偏好
- 审核 AI 从对话中识别出的记忆
- 归档过期记忆
- 标记“不要主动提”

新增字段建议：

```sql
memory_type     TEXT  -- key_fact / life / preference / avoid_topic
visibility      TEXT  -- ai_only / elder_visible / parent_only
active_policy   TEXT  -- proactive / background / never_proactive
created_by      TEXT  -- parent / dialog / system
review_status   TEXT  -- pending / approved / archived
```

### 5.3 记忆注入规则

| 类型 | 是否主动提 | 示例 |
|---|---|---|
| 用药/慢病/忌口 | 可以主动关照 | “今天药吃了吗？” |
| 生活偏好 | 只在自然相关时提 | “下午要不要听会儿评书？” |
| 敏感事件 | 默认不主动提 | 丧偶、住院、家庭矛盾 |
| 子女特别叮嘱 | 按策略执行 | “别总问孙子什么时候来” |

## 6. “成本”改为“充值”

### 6.1 命名调整

当前“本月关怀成本”应改为：

```text
充值与余额
```

底部导航：

```text
成本 → 充值
```

页面中避免出现“成本控制”“token 成本”等词。

### 6.2 子女端充值页结构

建议页面分区：

1. 当前余额
2. 可用陪伴时长
3. 本月使用
4. 充值套餐
5. 计费说明
6. 充值记录

示例文案：

```text
当前余额 ¥38.20
预计还可陪伴约 120 分钟
```

```text
本月已陪伴 18 次，共 96 分钟
已生成 12 次关怀分析
```

### 6.3 计费方式展示

不直接把技术 token 暴露给普通用户，但可以保留“明细”入口。

普通展示：

| 项目 | 展示方式 |
|---|---|
| 语音陪伴 | 按通话时长估算 |
| 关怀分析 | 按生成次数估算 |
| 声音角色训练 | 单次训练或套餐内次数 |

高级明细：

```text
语音服务、文本分析、角色训练由不同服务产生费用，实际扣费以账单明细为准。
```

### 6.4 数据表建议

```sql
CREATE TABLE wallet_accounts (
    elder_id TEXT PRIMARY KEY,
    balance_cents INTEGER NOT NULL DEFAULT 0,
    updated_at REAL NOT NULL
);

CREATE TABLE wallet_transactions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    elder_id TEXT NOT NULL,
    type TEXT NOT NULL,          -- recharge / consume / refund / grant
    amount_cents INTEGER NOT NULL,
    title TEXT NOT NULL,
    detail TEXT NOT NULL DEFAULT '',
    created_at REAL NOT NULL
);

CREATE TABLE billing_rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    item_code TEXT NOT NULL,     -- voice_minute / care_analysis / voice_clone
    display_name TEXT NOT NULL,
    unit TEXT NOT NULL,
    price_cents INTEGER NOT NULL,
    enabled INTEGER NOT NULL DEFAULT 1
);
```

### 6.5 API 建议

```http
GET /api/parent/{elder_id}/wallet
GET /api/parent/{elder_id}/wallet/transactions
GET /api/parent/{elder_id}/billing_rules
POST /api/parent/{elder_id}/recharge
```

MVP 阶段 `POST /recharge` 可以先做模拟充值：

```json
{
  "package_id": "care_50",
  "amount_cents": 5000
}
```

## 7. 后端架构改造

### 7.1 Character API 归属调整

当前角色 API 是：

```http
/api/character/{elder_id}
```

建议保留底层服务，但新增面向子女端的路由：

```http
/api/parent/{elder_id}/characters
```

老人端只读轻量接口：

```http
/api/elder/{elder_id}/companions
```

这样可以从 API 层体现职责边界。

### 7.2 角色表字段补充

建议 `characters` 增加：

```sql
created_by TEXT DEFAULT 'parent',
sync_status TEXT DEFAULT 'draft',
synced_at REAL,
elder_notice_seen_at REAL,
display_order INTEGER DEFAULT 0,
elder_alias TEXT DEFAULT '',
avoid_phrases TEXT DEFAULT '[]'
```

### 7.3 同步事件

新增角色同步事件表：

```sql
CREATE TABLE character_sync_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    elder_id TEXT NOT NULL,
    character_id INTEGER NOT NULL,
    event_type TEXT NOT NULL,       -- ready / activated / removed
    notice_text TEXT NOT NULL,
    seen_at REAL,
    created_at REAL NOT NULL
);
```

老人端读取后，如果 `seen_at IS NULL`，展示一次提示；点击后写回已读。

## 9. 推荐实施顺序

### 第一期：职责纠偏

1. 老人端底部“换个声音”改为“选择陪伴”。
2. 老人端角色页删掉创建、上传、训练表单。
3. 子女端新增“角色”卡片，迁移现有角色配置能力。
4. 角色 ready 后可以同步到老人端。

验收标准：

- 老人端看不到 speaker_id、上传录音、人格语料。
- 子女端可以创建角色并启用。
- 老人端可以选择已同步角色。

### 第二期：人文体验补强

1. 首页文案按时间和家庭事件动态变化。
2. 角色 ready 后老人端出现一次性温柔提示。
3. “聊过的事”改为“关心我的事”。
4. 所有技术词替换为家庭语言。

验收标准：

- 老人端无“克隆、训练、蒸馏、成本、token”等词。
- 子女端有明确的配置进度和授权提示。

### 第三期：充值与计费

1. 子女端“成本”改“充值”。
2. 增加钱包余额、充值记录、计费说明。
3. 保留 usage token 作为后台明细，不直接作为主展示。

验收标准：

- 子女端显示余额和可用时长。
- 可以看到充值记录和使用明细。
- 普通页面不直接暴露 token。

## 11. 最终目标体验

### 子女视角

```text
我给爸爸准备了女儿小芳的声音。
我上传了授权录音，补了几句小芳平时说话的方式。
我把爸爸最近膝盖疼、早上吃药这些事也写进去了。
同步后，爸爸手机上只会看到“小芳准备好了，要不要让她陪你聊聊”。
```

### 老人视角

```text
手机上没有复杂按钮。
我只看到小暖，或者女儿小芳。
想说话就点大圆圈。
小芳的声音准备好了，我愿意就选她，不愿意还用小暖。
```

### 产品视角

```text
技术仍然存在，但不站到老人面前。
老人得到的是陪伴，子女承担的是准备和照护。
这才是一个老年语音 Agent 应该有的端侧分工。
```
