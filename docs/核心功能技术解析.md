# 「小暖」核心功能技术解析

> 面向开发者的架构与实现归档。聚焦四大核心链路：**老人端对话**、**记忆蒸馏**、**历史记忆注入**、**子女端信号提炼与流转**。
> 缩写约定：WS = WebSocket（全双工长连接）；ASR = 自动语音识别；TTS = 语音合成；LLM = 大语言模型；VAD = 语音活动检测；PCM = 脉冲编码调制（原始音频）。

## 0. 整体架构速览

```
                  ┌──────────────── 老人端（web/elder） ─────────────────┐
   麦克风 PCM ───► │ app.js 采集 16k PCM ──► WS ──► 后端网关             │
   扬声器 ◄─────── │ player.js 播放 24k TTS ◄── WS ◄── 后端下行音频     │
                  └────────────────────────────────────────────────────┘
                                      │  /ws/elder/{elder_id}
                                      ▼
        ┌──────────────────── 后端 FastAPI（backend/server.py） ────────────────────┐
        │  ConversationSession（会话桥接，依赖倒置）                                  │
        │     ├─ VoiceEngine（seeduplex 达标 / fake 联调）── 火山豆包端到端实时语音   │
        │     ├─ context_provider() ── 记忆 + 人格 注入 system_prompt               │
        │     ├─ speaker_provider() ── 克隆音色 speaker_id 注入发声链路             │
        │     └─ on_session_end()  ── 会话结束异步触发 蒸馏 + 信号（不阻塞）         │
        │                                                                          │
        │  L3 MemoryService（记忆蒸馏/注入）   L4 SignalService（子女端信号）         │
        │  CharacterService（声音克隆+人格）   UsageStore（token 记账）              │
        └──────────────────────────────────────────────────────────────────────────┘
                                      │  SQLite（backend/memory.db）
                                      ▼
        ┌──────────────── 子女端（web/parent） ──── HTTP 拉取 ────────────────┐
        │  /api/parent/{elder_id}/signals  每日信号（结论，无原始对话）         │
        │  /api/parent/{elder_id}/key_facts 重点事项配置                       │
        └────────────────────────────────────────────────────────────────────┘
```

**技术栈**：FastAPI + aiosqlite（异步 SQLite）后端；原生 HTML/CSS/JS（AudioWorklet）前端；火山引擎豆包端到端实时语音（发声）+ 火山方舟 LLM（文本蒸馏/信号）。

**两套互不通用的鉴权体系**（见 [config.py](file:///Users/bytedance/Desktop/语音项目/backend/config.py)）：
- `VOLC_*`（openspeech 语音控制台）：负责「听 + 说」全双工发声 + 声音克隆训练。
- `ARK_API_KEY`（方舟大模型平台）：仅用于纯文本的记忆蒸馏与信号生成，**不参与发声**。

**隐私硬边界**：原始对话句子**从不落库**，只持久化「蒸馏后的结构化记忆」与「结论性信号」。

## 1. 老人端对话功能：实现流程与关键代码

### 1.1 设计目标
适老化「即点即说、可随时打断、亚秒延迟」的全双工语音陪伴。前端是瘦客户端，凭证只在后端持有，端侧不接触任何密钥。

### 1.2 端到端时序

```
老人点击大圆圈
   │
   ├─(1) precheckReady：GET /healthz 预检语音凭证 ── 缺凭证→友好提示，不进入卡死流程
   │
   ├─(2) 申请麦克风 getUserMedia（开启 AEC 回声消除）
   │      └─ AudioWorklet(capture-worklet.js) 采集 → 16k/mono PCM
   │
   ├─(3) openSocket：建立 WS /ws/elder/{elder_id}（8s 超时兜底）
   │
   ▼
后端 elder_ws：
   ├─(4) 创建 ConversationSession，注入 4 个回调（见下）
   ├─(5) session.start()：
   │       engine.connect() → 取 system_prompt(记忆+人格) → 取 speaker(克隆音色)
   │       → engine.start_session(...) 注入火山 → 回 {"status":"connected"}
   │
   ├─(6) 双向转发循环：
   │       上行：WS 二进制帧(PCM) → session.push_audio → engine.send_audio
   │       下行：火山 TTS 音频帧 → on_audio → WS 二进制帧 → player 播放
   │       字幕：ASR/Chat 文本 → on_text → WS {"type":"text",...} → 显示
   │       打断：火山 ASR_INFO(用户开口) → {"type":"barge_in"} → 前端停播
   │
   ▼
老人挂断 / 断开
   └─(7) session.stop → finish_session → _finalize → on_session_end(transcript)
            └─ 异步 distill（记忆蒸馏）+ generate（信号）—— 不阻塞
```

### 1.3 关键代码逻辑

**前端采集与播放**（[app.js](file:///Users/bytedance/Desktop/语音项目/web/elder/app.js)）：
- `start()` 串联四步：预检 → 麦克风（强制 `echoCancellation` 防 TTS 自播被回采误判打断）→ AudioWorklet 采集 → `openSocket()`。任一环节失败统一进 `failStart()` 收口（清理资源 + 适老化提示 + 恢复可重试）。
- `openSocket()` 用 **Promise 封装 + 8 秒超时**（[app.js#L150-L169](file:///Users/bytedance/Desktop/语音项目/web/elder/app.js#L150-L169)），避免凭证缺失时无限期卡在「正在接通…」——这是项目鲁棒性的关键约束。
- **本地 VAD 即时打断**（`localVad`）：Agent 出声期间连续 4 帧高能量（RMS ≥ 0.04）即判定老人插话，本地立刻停播，再配合「音频门闩 `acceptingAudio`」丢弃在途旧轮残尾，下一帧文本到达才重新开闸。这让打断「先于」服务端回包生效，体感更跟手。

**后端会话桥接**（[session/manager.py](file:///Users/bytedance/Desktop/语音项目/backend/session/manager.py)）：
- `ConversationSession` 职责单一——只做「客户端 ↔ 引擎」双向转发与生命周期管理，通过 4 个注入回调与上层解耦（依赖倒置）：
  - `context_provider()` → `(system_prompt, dialog_context)`：StartSession 前注入。
  - `speaker_provider()` → `speaker_id`：注入克隆音色。
  - `on_turn_text(role, text)`：逐轮字幕（当前为 None，转写在 `_transcript` 内累积）。
  - `on_session_end(transcript)`：会话结束触发异步蒸馏与信号。
- `_handle_text` 把每轮 ASR/Chat 文本累积进内存 `self._transcript`（**仅会话期内存，不落库**），同时下发字幕给前端。
- `_finalize()` 中 `on_session_end` 用 `try/except` 包裹：**蒸馏/信号失败绝不影响通话链路**。

**发声引擎抽象**（[engine/base.py](file:///Users/bytedance/Desktop/语音项目/backend/engine/base.py) + [factory.py](file:///Users/bytedance/Desktop/语音项目/backend/engine/factory.py)）：
- 统一 `VoiceEngine` 抽象，三回调 `on_audio / on_text / on_event`。
- `seeduplex`（[seeduplex.py](file:///Users/bytedance/Desktop/语音项目/backend/engine/seeduplex.py)）= 达标主路线，封装火山协议（[volc/client.py](file:///Users/bytedance/Desktop/语音项目/backend/volc/client.py)）：`StartConnection → StartSession → TaskRequest(音频流) ↔ TTS/ASR/Chat 回包 → FinishSession`。上行 16k、下行 24k，鉴权 Header 全部在后端注入。
- `fake`（[fake.py](file:///Users/bytedance/Desktop/语音项目/backend/engine/fake.py)）= 无凭证联调：脚本化模拟回包，并**真实反映注入的记忆与音色**，可在零凭证下端到端演示三大功能（通话/记忆/克隆）。引擎切换对上层零改动。

### 1.4 火山协议要点（[volc/client.py](file:///Users/bytedance/Desktop/语音项目/backend/volc/client.py)）
- `start_session` 把 `system_prompt` 写入 `dialog.system_role`，`speaker` 写入 `tts.speaker`，下行音频固定 `pcm_s16le / 24k`。
- `dialog.extra.input_mod = "keep_alive"` 静音保活，防音频流超时。
- `asr.end_smooth_window_ms = 1000` 适配老人语速的判停窗口（800–1200ms）。
- `receive_loop` 把回包分发：`ASR_RESPONSE(451)`→`on_text("user")`、`CHAT_RESPONSE(550)`→`on_text("assistant")`、`ASR_INFO(450)`→打断事件、`SESSION_FINISHED(152)`→退出循环。

## 2. 记忆蒸馏：算法逻辑、存储位置与数据结构

### 2.1 三层记忆模型（核心设计）

| 层级 | 名称 | 作用范围 | 由谁产生 | 是否落库 |
|------|------|----------|----------|----------|
| 会话内 | 引擎原生上下文 | 同一通话内连贯 | 火山引擎自身 | 否（引擎内存） |
| 跨会话-A | 重点事项 `key_facts` | 跨通话高优注入 | 子女预设 + 对话蒸馏 | ✅ SQLite |
| 跨会话-B | 生活记忆 `life_memories` | 跨通话近优注入 | 会话后 LLM 蒸馏 | ✅ SQLite |
| 隐私层 | 原始对话 | 仅会话内存 | — | ❌ **从不落库** |

> 关键点：会话内连贯交由引擎原生上下文，因此 `build_context` 返回的 `dialog_context` 恒为空数组，**不回填历史原句**——这是隐私边界的实现保证。

### 2.2 蒸馏算法逻辑（[memory/distiller.py](file:///Users/bytedance/Desktop/语音项目/backend/memory/distiller.py)）

触发点：会话结束 → `_on_session_end` → `asyncio.create_task(_memory.distill(elder_id, transcript))`（[server.py#L90-L97](file:///Users/bytedance/Desktop/语音项目/backend/server.py#L90-L97)），**异步、解耦、不阻塞**。

`distill()` 流程：
1. 空转录或方舟未启用（无 `ARK_API_KEY`）→ 直接跳过（降级为仅用已有记忆）。
2. `_format_dialogue` 把 `transcript` 渲染成「老人：… / 小陪：…」纯文本。
3. 调用方舟 LLM（`chat_json`），用固定 system prompt 要求**只输出 JSON**：
   ```json
   {
     "life_memories": ["近况/偏好/情绪/家常，每条一句话"],
     "key_facts": [{"category": "用药|慢病|忌口|重要日期|紧急联系人|其他", "content": "一句话"}]
   }
   ```
4. `_persist` 落库：生活记忆逐条写 `life_memories`；重点事项校验 `category` 合法性（非法归「其他」）后写 `key_facts`，来源标 `source="dialog"`。

**降级原则**：方舟不可用或返回非法 JSON → 跳过本轮蒸馏，下次仍用已有记忆，**绝不中断语音通话**（[text_client.py](file:///Users/bytedance/Desktop/语音项目/backend/ark/text_client.py) 的 `_safe_json` 容错剥离 ```` ```json ```` 围栏）。

### 2.3 存储位置、格式与数据结构

- **物理位置**：单个 SQLite 文件，默认 `backend/memory.db`（由 `DB_PATH` 环境变量可覆盖，见 [config.py#L80](file:///Users/bytedance/Desktop/语音项目/backend/config.py#L80)）。所有模块（记忆/信号/角色/用量）共用该文件，已通过 `.gitignore` 排除，严禁提交。
- **访问方式**：`aiosqlite` 异步驱动，每个方法独立开连接（避免长连接跨协程竞争）。

**表 `key_facts`（层级 A 重点事项）**（[memory/store.py#L19-L37](file:///Users/bytedance/Desktop/语音项目/backend/memory/store.py#L19-L37)）：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增 |
| elder_id | TEXT | 老人标识（物理隔离键） |
| category | TEXT | 枚举：用药/慢病/忌口/重要日期/紧急联系人/其他 |
| content | TEXT | 一句话内容 |
| source | TEXT | `parent`=子女预设 / `dialog`=对话识别 |
| updated_at | REAL | 时间戳 |
| — | UNIQUE | `(elder_id, category, content)` 去重；冲突时 `UPSERT` 更新来源与时间 |

**表 `life_memories`（层级 B 生活记忆）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增 |
| elder_id | TEXT | 老人标识 |
| content | TEXT | 一句话生活记忆 |
| created_at | REAL | 时间戳 |
| — | UNIQUE | `(elder_id, content)`，`INSERT OR IGNORE` 去重 |
| — | INDEX | `(elder_id, created_at DESC)` 支撑近优读取 |

## 3. 新一轮对话：历史记忆的读取与注入流程

### 3.1 完整链路

```
新通话建立 WS → session.start()
   │
   ▼
context_provider() = _build_context_with_persona(elder_id)   [server.py#L100-L114]
   │
   ├─ (A) MemoryService.build_context(elder_id)              [distiller.py#L62-L86]
   │       ├─ store.list_key_facts(elder_id)        ← 读层级 A（按 updated_at DESC）
   │       ├─ store.recent_life_memories(elder_id, limit=15) ← 读层级 B（按时间近优取 15 条）
   │       └─ 拼装 system_prompt（基础人设 + 重点事项块 + 近期记忆块）
   │
   ├─ (B) CharacterService.active_persona(elder_id)          ← 启用角色的人格提示词（就绪才返回）
   │       └─ 拼到 system_prompt 末尾（「第二蒸馏」人格注入点）
   │
   ▼
engine.start_session(system_prompt=..., dialog_context=[], speaker=克隆音色)
   └─ 火山写入 dialog.system_role → 模型据此「记得之前聊过的事」并主动关照
```

### 3.2 system_prompt 拼装结构（[distiller.py#L79-L86](file:///Users/bytedance/Desktop/语音项目/backend/memory/distiller.py#L79-L86)）

```
你是老人的语音陪伴伙伴，名字叫小陪……（基础人设 _BASE_PERSONA）

【需要特别记住并主动关照的重点事项】
- [用药] 每天早晚各一次降压药
- [慢病] 有高血压

【最近聊到的事，可自然提起以示在乎】
- 膝盖最近有点疼
- 上周孙子来看过他

【你要扮演的角色设定……】        ← 若启用了角色（人格蒸馏产出），拼接在最后
- 角色：女儿小芳（女儿）
- 语气：…  - 说话习惯：…
```

### 3.3 设计要点
- **高优先级注入**：层级 A 重点事项放在前部，确保模型优先关照（用药/慢病等关键信息）。
- **近优截断**：层级 B 只取最近 15 条，控制 prompt 体积与成本。
- **隐私保证**：注入内容**只来自蒸馏后的结构化记忆**，`dialog_context` 恒空，绝不读取/拼接任何原始对话原句。
- **容错**：读记忆失败 → 回落 `_BASE_PERSONA` 空记忆，通话照常进行。
- **可验证**：在 fake 引擎下，注入了历史记忆时首轮会主动说「我记得您之前说过 X，今天感觉怎么样？」（[fake.py#L81-L87](file:///Users/bytedance/Desktop/语音项目/backend/engine/fake.py#L81-L87)），直观演示跨会话连贯。

## 4. 信息提炼与子女端流转

### 4.1 系统提炼哪些信息

会话结束后并行启动两条提炼链路（互不干扰）：

**(A) 记忆蒸馏**（给 AI 用，见第 2 节）：生活记忆 + 重点事项。

**(B) 子女端信号**（给子女看，[signals/rules.py](file:///Users/bytedance/Desktop/语音项目/backend/signals/rules.py)）：规则引擎对**仅老人说的话**做关键词检测，提炼为「结论性标签」，**绝不含原始句子**：

| 维度 | 取值 | 提炼规则 |
|------|------|----------|
| `level` 等级 | normal / attention / urgent | 命中话题关键词，按最高优先级升级；紧急话题命中即 urgent |
| `mood` 心情 | 积极 / 平稳 / 低落 | 积极词 vs 负面话题词计数对比；urgent 直接判低落 |
| `mentions` 提及 | `[{topic, count}]` | 话题标签 + 出现次数 |
| `alerts` | 话题展示名列表 | attention/urgent 级话题 |

话题规则库（关键词→分类→等级）：
- **紧急情况**（urgent）：摔倒/跌倒/救命/喘不上气/胸口疼/晕倒/出血…
- **身体不适**（attention）：疼/痛/头晕/发烧/咳嗽/没力气…
- **情绪低落**（attention）：孤独/想哭/心烦/失眠/没胃口…
- **用药提醒**（attention）：忘了吃药/没吃药/停药…

### 4.2 触发条件与处理流程

```
on_session_end(transcript)
   └─ asyncio.create_task(_signals.generate(elder_id, transcript))   [server.py#L94-L95]
        │  触发条件：① 会话正常结束且 ② transcript 非空
        ▼
   SignalService.generate()                       [signals/engine.py#L85-L96]
        ├─ rules.analyze_text(transcript)  ← 规则提炼结论结构
        ├─ _summarize()                    ← 方舟 LLM 润色成 30 字内温暖摘要
        │     └─ 方舟不可用 → 降级用 rules.build_summary() 规则摘要
        └─ _persist()                      ← 写入 signals 表（含 mood_score 数值化）
```

> **重要：当前是「拉取式」而非服务端主动推送。** 后端会话结束后异步把信号**落库**；子女端前端（[web/parent/index.html](file:///Users/bytedance/Desktop/语音项目/web/parent/index.html)）打开页面时通过 `fetch` `GET /api/parent/{elder_id}/signals` 拉取展示。即「提炼即落库 → 子女端按需读取」。如需真正的实时推送（如 urgent 告警），需另接 WS/推送通道，目前尚未实现。

### 4.3 子女端数据格式

**信号存储表 `signals`**（[signals/engine.py#L24-L36](file:///Users/bytedance/Desktop/语音项目/backend/signals/engine.py#L24-L36)）：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增 |
| elder_id | TEXT | 老人标识 |
| level | TEXT | normal/attention/urgent |
| mood | TEXT | 积极/平稳/低落 |
| mood_score | INTEGER | 数值化：积极+1/平稳0/低落-1（供情绪趋势序列） |
| mentions | TEXT(JSON) | `[{"topic","count"}]` |
| summary | TEXT | 一句温暖摘要（仅结论，无原话） |
| created_at | REAL | 时间戳 |

**接口响应格式** `GET /api/parent/{elder_id}/signals`（[api/parent.py#L68-L70](file:///Users/bytedance/Desktop/语音项目/backend/api/parent.py#L68-L70)）：
```json
{
  "items": [
    {
      "level": "attention",
      "mood": "低落",
      "mentions": [{"topic": "情绪低落", "count": 2}],
      "summary": "今日情绪低落；提到情绪低落2次",
      "created_at": 1719500000.0
    }
  ]
}
```

**子女端的两类接口**（严格隔离，[api/parent.py](file:///Users/bytedance/Desktop/语音项目/backend/api/parent.py)）：
1. **重点事项配置**（写入层级 A，`source="parent"`）：`GET/POST/DELETE /api/parent/{elder_id}/key_facts` —— 子女预设的会反向参与下次对话的记忆注入。
2. **信号查看**（只读结论）：`GET /api/parent/{elder_id}/signals`、`GET /api/parent/{elder_id}/usage`（成本看板，仅 token 计数）。

### 4.4 隐私硬边界（贯穿全链路）
- `signals` 表只存等级/心情/标签/次数/摘要，**绝不写入任何原始对话句子**；摘要文本也只描述结论、不复述原话（LLM system prompt 强制约束）。
- 子女端路由**只暴露结论性信息**，绝无原始对话接口。
- `life_memories`（含家常细节）属「老人看自己的数据」，仅老人端 `GET /api/elder/{elder_id}/memories` 可读，**绝不进子女端路由**，二者严格分离（[api/elder.py](file:///Users/bytedance/Desktop/语音项目/backend/api/elder.py)）。

## 5. 附：角色再生（声音克隆 + 人格蒸馏）

虽非四大主题，但与对话/记忆链路深度耦合，简述如下：

- **一个角色 = 声音（speaker_id）+ 灵魂（persona_prompt）**，两条蒸馏链路独立推进（[character/service.py](file:///Users/bytedance/Desktop/语音项目/backend/character/service.py)）：
  - **声音克隆**：上传音频 → 火山声音复刻 API（ICL2.0）训练 `speaker_id` → `training → ready` 状态轮询（[voice_clone.py](file:///Users/bytedance/Desktop/语音项目/backend/character/voice_clone.py)，无凭证时 mock 6 秒模拟流转）。
  - **人格蒸馏（第二蒸馏）**：上传角色语料 → 方舟 LLM 蒸馏五层人格 JSON（身份/语气/说话习惯/知识背景/价值取向 + 禁忌）→ 渲染成 system prompt 片段（[character/persona.py](file:///Users/bytedance/Desktop/语音项目/backend/character/persona.py)）。
- **会话注入**：`active_speaker()` 注入 `tts.speaker` 决定「用谁的嗓子说」；`active_persona()` 拼到 system_prompt 末尾决定「用谁的口吻说」。未启用/未就绪均回落默认音色与基础人设，绝不阻塞通话。
- **隐私**：原始音频与语料仅用于训练/蒸馏，**用完即弃不入库**；DB 只存 `speaker_id` 与人格提示词。多角色按 `elder_id` 物理隔离，激活互斥。
